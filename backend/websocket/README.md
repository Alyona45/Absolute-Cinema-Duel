# WebSocket/Room protocol (actual contract)

## Connection

```
ws://<host>/ws/{room_id}/{user_id}?token=<JWT_ACCESS_TOKEN>
```

- `room_id` ŌĆö invite code of game session (or runtime room id for guest-only room).
- `user_id` ŌĆö participant id returned by room HTTP endpoints.
- `token`:
	- required for authenticated participants;
	- optional for guests.

### Close codes

| Code | Reason |
|------|--------|
| 4001 | Invalid/missing token for authenticated participant, token/user mismatch, or DB membership mismatch |
| 4004 | Room or participant not found |

## HTTP endpoints (rooms)

All endpoints are served by `backend/routers/ws.py`.

| Method | URL | Auth | Description |
|-------|-----|------|-------------|
| POST | `/rooms` | Optional | Create room. If authorized, creates DB-backed game session and room id = invite code. If guest, creates in-memory room. |
| POST | `/rooms/{room_id}/join` | Optional | Join room. Guests are allowed only for pure in-memory rooms; guest join into DB game sessions is denied. |
| POST | `/rooms/{room_id}/start` | Optional | Start room. For DB-backed room only authenticated host can start. |
| GET | `/rooms/{room_id}` | No | Get current runtime room state. |

Request body for `POST /rooms` and `POST /rooms/{room_id}/join`:

```json
{
	"username": "optional"
}
```

- For guests, `username` is used (or autogenerated `ąōąŠčüčéčī_<suffix>`).
- For authenticated users, username/email comes from account.

## Client ŌåÆ server events

Only one message type is currently supported:

| type | Payload | Description |
|------|---------|-------------|
| `PING` | ŌĆö | Keepalive/ping |

Invalid JSON or invalid message schema returns `ERROR` and keeps socket alive.

## Server ŌåÆ client events

| type | Payload | When |
|------|---------|------|
| `ROOM_STATE` | `{host_user_id, participants, status}` | Immediately after successful WS connect |
| `PLAYER_JOINED` | `{user_id, username, is_guest}` | New participant connected |
| `PLAYER_DISCONNECTED` | `{user_id}` | Participant disconnected |
| `GAME_STARTED` | `{room_id}` | Host started game |
| `PONG` | ŌĆö | Reply to `PING` |
| `ERROR` | `{message}` | Invalid incoming message |

## DB-backed room rules (must-know)

For authenticated/game-session rooms, start is allowed only if all checks pass:

1. Caller is the session host.
2. Session is in `CREATED` state.
3. Session has at least 2 participants.
4. Every participant has selected a movie (`selected_session_movie_id` is set).
5. Every selected movie belongs to the same session.

If any check fails, `POST /rooms/{room_id}/start` returns `4xx` with details.

## Runtime recovery

If process memory was reset but DB session exists, runtime room is reconstructed on demand by invite code:

- host restored from session owner;
- participants restored from session participants;
- runtime status synced from DB (`CREATED`/`PLAYING`/`FINISHED`/`CANCELLED`).

## Lifecycle and cleanup

`backend/main.py` starts background cleanup every 60 seconds.

Room is removed when both are true:

1. All participants are disconnected.
2. No activity for more than 60 seconds.
